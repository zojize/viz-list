# Viz List Revival — Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Revive and modernize viz-list — a C++ data structure visualizer — with updated infrastructure, a real execution engine, richer visualization, and a redesigned UI.

**Architecture:** Four-phase approach: (1) Infrastructure modernization — bun, latest deps, knip; (2) Interpreter/execution engine — explore WASM-based C/C++ execution in the browser; (3) Visualization — replace Cytoscape with a stack/heap memory model view with detailed data inspection; (4) UI redesign.

**Tech Stack:** Vue 3, Vite 8, TypeScript 6, Bun, UnoCSS, Monaco Editor, Tree-Sitter, Cytoscape (to be replaced)

## Phase 2: Interpreter / Execution Engine (Exploration Required)

> This phase requires research and prototyping before concrete tasks can be defined.

### Goals

- Replace the current JS-based AST interpreter with real C/C++ execution
- Maintain step-through debugging capability for visualization
- Run entirely in the browser (no backend)

### Approaches to Explore

1. **Instrument-then-compile**: Transform C++ source to inject `__viz_step()` callbacks at every statement, then compile instrumented code to WASM. Callbacks yield control back to JS for visualization. Best correctness, most work.

2. **C interpreter compiled to WASM**: Use an existing C interpreter (picoc, cparse, TCC in interpreter mode) compiled to WASM. Real-ish C semantics with natural step-through. Pragmatic middle ground.

3. **Clang/Emscripten in browser**: Full compiler toolchain running in WASM. Heavy download (~30MB+), but maximum correctness. Could lazy-load.

4. **Hybrid**: Keep Tree-Sitter for parsing/highlighting, hand off execution to WASM-based interpreter with real memory model.

### Key Constraint

Step-through debugging with visualization requires hooking into execution at each statement. Pure WASM compilation loses this — the instrumentation or interpreter approach is needed to pause and inspect state.

### Research Tasks

- [ ] Evaluate picoc compiled to WASM — does it support enough C++ (structs, pointers, new/delete)?
- [ ] Prototype the instrument-then-compile approach — how complex is the source transformation?
- [ ] Benchmark WASM compiler download sizes for each approach
- [ ] Determine if the current Tree-Sitter parser can be reused for any approach
- [ ] If staying with interpretation: catalog unsupported C++ features and prioritize which to add

## Phase 3: Visualization Overhaul

> Depends on Phase 2 decisions (execution engine determines what state is available to visualize).

### Goals

- Replace Cytoscape graph view with a stack/heap memory visualization
- Show actual memory layout: stack frames with locals, heap blocks with typed data
- Pointer arrows between memory locations
- Click any memory slot to see detailed view of data stored there
- Special visualizations for specific types (linked lists, trees, arrays)
- During simulation playback, auto-show detailed view of the action taking place

### Design Concept

```
+-------------------+  +---------------------------+
|   Stack           |  |   Heap                    |
|                   |  |                           |
| main()            |  |  [0x08] Node              |
|   head: 0x08 ---------> data: 5                  |
|   tail: 0x20 ---+ |  |  next: 0x14 ----------+  |
|   curr: 0x14 -+ | |  |                       |  |
|               | | |  |  [0x14] Node <---------+  |
|               | | |  |    data: 10               |
|               +-------->  next: 0x20 --------+   |
|                 | |  |                       |   |
|                 | |  |  [0x20] Node <--------+   |
|                 +------> data: 15                |
|                   |  |   next: NULL              |
+-------------------+  +---------------------------+

        [ Detail View: Node at 0x14 ]
        +---------------------------+
        | struct Node               |
        |   data: int = 10          |
        |   next: Node* = 0x20  ------>
        |   prev: Node* = 0x08  <------
        +---------------------------+
```

### Rendering Options to Evaluate

- **Canvas** (Pixi.js / Konva) — full control, good perf for custom layouts
- **SVG** (D3 or hand-rolled) — easier to style, natural click targets
- **HTML/CSS** (divs + arrows via SVG overlay) — simplest, most accessible, works with Vue reactivity natively
- **Keep Cytoscape** only for optional "graph overview" of linked lists / trees as a secondary view

### Research Tasks

- [ ] Prototype stack/heap layout with HTML/CSS + SVG arrow overlay
- [ ] Evaluate if D3 or a lighter library is needed for pointer arrow rendering
- [ ] Design the detail view component and its interaction model
- [ ] Define the data model for memory state that the execution engine exposes
- [ ] Plan special visualizations for linked lists, trees, arrays
